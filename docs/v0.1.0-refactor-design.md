# litxr v0.1.0 Refactor Design

This note proposes a structural refactor for `litxr v0.1.0`.

The goal is not to change the package purpose. The goal is to keep the current
user-facing workflows while replacing the internal storage model with one that
is:

- identity-first
- thinner on `fst`
- more explicit about relationships
- faster to update at 100k+ references
- easier to reason about when one work has multiple external ids

This note assumes the package will remain:

- local-first
- `ref_id`-friendly at the surface
- `data.table` + `fst` + JSON based
- non-R6
- non-Shiny

## Problem Summary

Current pain points come from denormalized storage and identity ambiguity.

Main issues:

- the same work can appear as both `arxiv:...` and `doi:...`
- relationship logic is encoded as row-merging heuristics instead of a stable
  identity layer
- collection and project indexes store too much mutable row payload
- some updates rewrite large `fst` files when only small status changes are
  needed
- keyed lookups depend on indexes that are larger and more fragile than needed
- rich fields and operational status are mixed into the same row-shaped stores

This causes:

- slow updates
- hard-to-debug sync behavior
- awkward DOI/arXiv transitions
- more I/O than necessary

## Design Goals

`litxr v0.1.0` should:

1. introduce a stable internal identity key per work
2. treat arXiv ids, DOIs, ISBNs, and other ids as identities
3. keep `fst` files small and operational
4. keep rich metadata and LLM digests in JSON
5. preserve current surface behavior for most user-facing commands
6. make full rebuilds rare and explicit

## Target Data Model

### 1. Internal identity

Introduce a package-internal key:

- `entity_id`

Rules:

- one `entity_id` represents one conceptual work
- one `entity_id` may have many external identifiers
- `entity_id` is stable even if preferred citation form changes over time

Examples:

- one work may have:
  - `arxiv:2505.10468`
  - `doi:10.1016/j.inffus.2025.103599`
- both should map to the same `entity_id`

### 2. External id identity table

Add a thin identity index:

- `index/ref_identity_map.fst`

Suggested columns:

- `ref_id`
- `entity_id`
- `id_type`
- `source`
- `source_id`
- `is_primary_ref_id`
- `is_published_form`
- `linked_at`
- `status_code`

Purpose:

- resolve any input `ref_id` to `entity_id`
- support multiple equivalent identifiers for one work
- stop relying on record-merging heuristics for identity

### 3. Thin entity index

Add a compact entity projection:

- `index/entities.fst`

Suggested columns:

- `entity_id`
- `primary_ref_id`
- `preferred_citation_ref_id`
- `display_title`
- `year`
- `first_author`
- `author_short`
- `entry_type`
- `primary_source`
- `published_source`
- `has_md`
- `has_llm_digest`
- `has_embeddings`
- `has_published_metadata`
- `updated_at`

Important rule:

- this table is not the place for full bibliographic payload
- it is a search and status projection only

### 4. Collection membership index

Keep a thin membership table:

- `index/entity_collections.fst`

Suggested columns:

- `entity_id`
- `collection_id`
- `recorded_at`

This replaces the need to join membership on duplicated row payload.

### 5. Artifact status index

Add or reshape a status table:

- `index/entity_status.fst`

Suggested columns:

- `entity_id`
- `has_ref_json`
- `has_fulltxt_md`
- `has_llm_digest`
- `has_standardized_findings`
- `has_descriptive_stats`
- `has_anchor_references`
- `has_citation_logic_nodes`
- `has_embedding_abstract`
- `has_embedding_summary`
- `digest_schema_version`
- `digest_revision_latest`
- `updated_at`

This should be the main operational status table.

### 6. Rich payloads stay in JSON

Keep JSON as the durable store for rich detail.

Per-entity or per-identity JSON should store:

- full bibliographic payload from source
- arXiv raw payload
- Crossref raw payload
- digest JSON
- findings JSON if needed
- table extraction payloads

The key design rule is:

- `fst` for relationships and search projections
- JSON for full payloads

### 7. Separate source records from canonical entity state

Each source contribution should remain traceable.

Proposed durable folders:

- `records/arxiv_ref_json/`
- `records/crossref_ref_json/`
- `records/manual_ref_json/`
- `digests/`
- `md/`

And one canonical lightweight entity projection in `fst`.

That means:

- source payloads are not overwritten by later published forms
- canonical “best available citation” is determined by identity/entity rules

## Ref Resolution Rules

### 1. Surface behavior

Surface functions may continue to accept:

- `ref_id`
- DOI strings
- bare arXiv ids

But internally the resolution order becomes:

1. normalize input into canonical identity form
2. look up `entity_id` in `ref_identity_map.fst`
3. use `entity_id` for status and relationship operations
4. choose `ref_id` projection based on task policy

### 2. Policy examples

For BibTeX export:

- prefer `preferred_citation_ref_id`
- usually DOI-backed published record if richer

For digest prompt building:

- if requested `ref_id` is DOI and linked arXiv exists, prefer arXiv full-text
  hints
- but preserve the requested DOI context in metadata

For collection searches:

- resolve matched identities back to `entity_id`
- return either identity rows or canonical entity rows depending on API contract

## Storage Layout Proposal

### New core `fst` files

Keep these as the main operational indexes:

- `index/ref_identity_map.fst`
- `index/entities.fst`
- `index/entity_collections.fst`
- `index/entity_status.fst`
- `log/<collection>_collection_fetch_history.tsv`

Optional specialized tables remain valid:

- `findings/standardized_findings.fst`
- `findings/descriptive_statistics.fst`
- `findings/anchor_references.fst`
- `findings/citation_logic_nodes.fst`

### Files that should shrink

Current row-shaped indexes such as:

- collection `index/references.fst`
- project `index/references.fst`

should stop storing:

- long author arrays
- abstracts
- raw source payload fragments
- rich digest-like fields
- repeated descriptive bibliographic detail when not needed for search

They should become thin searchable projections only.

## Which Current `fst` Files Should Disappear Or Shrink

### Should disappear

These should be replaced conceptually:

- `project.data_root/index/reference_collections.fst`
  - replace with `entity_collections.fst`
- any future direct arXiv/DOI equivalence heuristics embedded only in row merges
  - replace with `ref_identity_map.fst`

### Should shrink substantially

These should become thin projections:

- collection-level `index/references.fst`
- collection-level `index/references_delta.fst`
- project-level `index/references.fst`

They should keep only:

- identity keys
- entity key
- minimal search fields
- light status fields

They should not keep:

- full abstract text unless explicitly justified as a search projection
- nested list payloads
- LLM digest fields
- repeated large strings that are already available in JSON

### Should stay

These remain useful with limited change:

- embedding metadata `fst` stores
- findings tables that are already purpose-built and narrow

## Migration Plan

Migration should be explicit and staged.

### Phase 0. Freeze old semantics

Before migration:

- define current compatibility behaviors
- define which current functions must remain stable
- define default preference rules for DOI vs arXiv

### Phase 1. Build identity and entity indexes from existing stores

Create a one-time migration builder:

- scan current collection `ref_json`
- derive candidate `entity_id`
- populate `ref_identity_map.fst`
- populate `entities.fst`
- populate `entity_collections.fst`
- populate `entity_status.fst`

Initial heuristic:

- if arXiv row has a DOI and a DOI row exists, join them to one `entity_id`
- otherwise keep one identity per entity

This phase should not rewrite source JSON.

### Phase 2. Introduce compatibility lookup layer

Update lookup functions so they resolve:

- `ref_id` -> `entity_id`
- `entity_id` -> preferred identity row

Keep current public function names where possible.

### Phase 3. Rewrite write paths

Update ingest and enrich paths so they:

- write source payload JSON
- update identity/entity/status indexes only
- do not rewrite large denormalized project reference rows

This is where most performance improvement will come from.

### Phase 4. Migrate LLM and findings linkage

Move digest and downstream artifacts to `entity_id`-aware linkage while still
supporting current `ref_id` surface calls.

Recommended compatibility rule:

- store digest file names by requested surface `ref_id` only if needed for
  backward compatibility
- but status and relationship indexes should be `entity_id`-based

### Phase 5. Deprecate old heavy reference indexes

After compatibility is proven:

- deprecate direct dependence on old project `references.fst`
- deprecate row-merge identity heuristics
- keep read-only migration helpers for older stores

## Compatibility Layer For Existing Scripts

The refactor should preserve practical workflows.

### 1. `litxr_find_refs()`

Keep surface API.

New behavior:

- resolve input identities to `entity_id`
- return identity-facing rows shaped like today
- prefer primary or task-specific identity when multiple identities exist

### 2. `write_bib_by_ref_ids.sh`

Keep the same CLI.

Internally:

- resolve each input `ref_id` to `entity_id`
- choose `preferred_citation_ref_id`
- load the JSON/source payload of that preferred identity
- generate BibTeX from that richer representation

### 3. digest prompt and ingest scripts

Keep the same script names and parameters where possible.

New logic:

- if the requested `ref_id` is DOI and linked arXiv identity exists, prompt builder
  uses the linked arXiv full-text hints
- digests remain associated with the requested surface `ref_id` for user
  transparency, but entity-level status should still be updated

### 4. collection repair scripts

Keep scripts like:

- `repair_arxiv_range.R`
- `repair_arxiv_latest.R`

But make them update:

- source payload JSON
- identity/entity/status indexes

instead of repeatedly rewriting fat collection reference rows.

### 5. human-facing summary scripts

Keep human scripts unchanged at the CLI level.

They should read:

- status/index data from thin `fst`
- rich fields from JSON only for selected entities

This should make them more deterministic and easier to debug.

## Performance Model

### Current anti-pattern

The current anti-pattern is:

- read large row stores
- merge many repeated fields
- rewrite broad `fst` payloads after small updates

### Target pattern

The target pattern is:

- point lookup in thin identity/entity/status indexes
- read only selected JSON files for rich detail
- update a few small `fst` tables when status changes
- keep rebuilds explicit and rare

Expected effects:

- faster DOI/arXiv linking
- faster status updates
- smaller `fst` files
- lower I/O during incremental ingest
- less corruption impact because indexes are thinner and rebuild scope is
  clearer

## Identity Rules For DOI And arXiv

This should be explicit in `v0.1.0`.

Rules:

- one work can have both arXiv and DOI identities
- `entity_id` is the real identity
- BibTeX export usually prefers DOI-backed identity
- digest/full-text workflows may prefer linked arXiv identity for retrieval
- no user-facing script should need to know the internal join logic

This is a cleaner realization of the current intended behavior.

## Non-Goals

This refactor should not:

- introduce R6 reference objects
- introduce a database dependency
- host AI agents inside the package
- move to Shiny or another UI layer
- break the current `ref_id`-centric user workflows without strong reason

## Suggested `v0.1.0` Milestones

1. Add `entity_id` and `ref_identity_map.fst`.
2. Build `entities.fst`, `entity_collections.fst`, and `entity_status.fst`.
3. Make DOI/arXiv linking identity-based.
4. Shrink collection/project reference indexes to thin projections.
5. Route BibTeX, digest, and summary scripts through identity/entity resolution.
6. Add explicit migration and audit helpers.

## Recommended Implementation Order

For risk control, the safest sequence is:

1. introduce new indexes alongside old ones
2. build migration helpers and validators
3. switch read paths first
4. switch write paths second
5. retire old heavy indexes last

This avoids a big-bang migration.

## Planned Version Roadmap

The package is currently on the `0.0.8.x` line. The roadmap below keeps that
line for incremental compatibility-preserving refactor work, then uses
`0.1.0` only when the entity-first architecture is truly the primary internal
model.

The principle is:

- `0.0.8.x` means staged internal migration with stable user-facing workflows
- `0.1.0` means the refactor is complete enough to be the declared package
  architecture

### `0.0.8.16`

Goal:

- finish the remaining reader and read-policy migration onto the shared
  identity/entity resolver

Main targets:

- move the remaining human-facing lookup and report scripts onto shared
  resolver helpers
- stop duplicating DOI/arXiv normalization logic across scripts where that
  duplication is no longer necessary
- make digest/report readers consistently choose task-policy identities:
  - published identity for citation-oriented output
  - linked arXiv identity for digest/full-text retrieval when that is the better
    source

Done means:

- no important user-facing report/export/prompt script performs its own ad hoc
  identity guessing
- the package has one clearly reusable resolver layer for:
  - input normalization
  - identity lookup
  - preferred citation identity selection
  - preferred digest/full-text identity selection
- CLI behavior remains unchanged for existing scripts

### `0.0.8.17`

Goal:

- shrink collection and project reference projections so they stop carrying
  unnecessary repeated payload

Main targets:

- define the minimal searchable projection contract for:
  - collection `index/references.fst`
  - collection `index/references_delta.fst`
  - project `index/references.fst`
- remove or reduce heavy repeated fields that are already owned by JSON
- keep only what is needed for:
  - exact lookup
  - lightweight search
  - display-level metadata
  - operational joins

Done means:

- collection/project reference projection files are materially thinner
- large repeated payload such as rich nested content or unnecessarily long
  text is no longer stored in projection indexes unless explicitly justified
- existing callers still receive compatible output shapes

### `0.0.8.18`

Goal:

- finish the write-path demotion of fat reference indexes

Main targets:

- make ingest, enrich, add-record, and repair paths update:
  - source JSON
  - thin identity/entity indexes
  - thin status indexes
  - thin projection caches
- stop treating project `references.fst` as the primary mutable semantic store
- make damaged projection indexes cheap and explicit to rebuild from
  authoritative state

Done means:

- normal incremental ingest no longer depends on broad row-shaped reference
  rewrites
- source JSON plus thin entity indexes are the practical authoritative store
  for new writes
- projection caches can be rebuilt without ambiguity about what is canonical

### `0.0.8.19`

Goal:

- complete entity-first artifact linkage and operational status reporting

Main targets:

- standardize artifact/status linkage on `entity_id`
- make coverage and enrichment status views depend primarily on
  `entity_status.fst`
- tighten audits for:
  - stale status rows
  - orphan artifacts
  - digest revision mismatches
  - identity/entity inconsistencies

Done means:

- operational status reporting is entity-based end to end
- readers no longer rely on full status-table rebuilds during normal use
- stale-cache versus authoritative-artifact mismatches are easy to detect and
  report

### `0.0.8.20`

Goal:

- release-hardening for the `0.1.0` transition

Main targets:

- add explicit migration helpers and operator-facing diagnostics for older
  stores
- document stable identity policies for:
  - citation export
  - digest/full-text prompting
  - arXiv/DOI linking
  - collection membership behavior
- verify that legacy CLI workflows still behave correctly under the new
  internal model

Done means:

- users do not need to understand `entity_id` to keep using existing scripts
- migration/audit tooling is explicit enough to diagnose stale or mixed stores
- package docs explain the new internal model clearly enough for maintenance
  and extension work

### `0.0.8.21`

Goal:

- make entity/identity indexes the primary generic read-resolution layer

Main targets:

- route `litxr_find_refs()` exact-key and filtered lookup through:
  - `ref_identity_map.fst`
  - `entities.fst`
  - `entity_collections.fst`
- stop using project `references.fst` as the first semantic lookup surface for:
  - bare `ref_id`
  - DOI/arXiv identity resolution
  - collection membership filtering
- introduce one explicit hydration step that loads rich projection rows only
  after candidate identity/entity resolution is done

Done means:

- `litxr_find_refs()` is entity-first internally while preserving its current
  return shape
- collection scoping no longer depends primarily on
  `reference_collections.fst`
- the package has one clear separation between:
  - candidate resolution
  - row hydration
  - user-facing output shaping

### `0.0.8.22`

Goal:

- demote old project reference caches to compatibility projections only

Main targets:

- make `index/references.fst` and `index/reference_collections.fst` clearly
  rebuildable compatibility views rather than the semantic center of the
  package
- ensure identity/entity indexes can be refreshed from authoritative collection
  state without requiring those project caches to be healthy first
- tighten read-path fallbacks so damaged projection caches degrade into:
  - thin-index rebuild
  - targeted hydration
  - explicit warnings
  instead of broad merged-table dependence

Done means:

- old project reference caches are operationally optional for core semantics
- rebuild direction is unambiguous:
  - authoritative collection state
  - thin identity/entity/status indexes
  - compatibility projections last
- package internals no longer treat old project caches as the source of truth

### `0.0.8.23`

Goal:

- release-hardening for the actual `0.1.0` architecture milestone

Main targets:

- split large internal code surfaces, especially `R/sync.R`, into focused files
  such as:
  - reference projections and caches
  - identity/entity indexes
  - entity status and audits
  - repair/migration helpers
- add focused tests that prove the architecture boundary:
  - authoritative JSON ownership
  - thin-index-first reads
  - compatibility projection rebuilds
  - DOI/arXiv identity stability
- review package docs and CLI help text for any remaining projection-first
  wording

Done means:

- the codebase structure matches the new architecture well enough that future
  maintenance does not depend on navigating one monolithic `sync.R`
- tests describe the new architecture directly, not only user-facing behavior
- the remaining jump to `0.1.0` is a release declaration, not another major
  internal rewrite

### `0.1.0`

Goal:

- declare the identity-first refactor complete

Main targets:

- make the identity/entity layer the real primary internal architecture
- ensure legacy heavy reference indexes are clearly demoted to thin
  projections or compatibility caches
- remove dependence on row-merge heuristics for DOI/arXiv identity
- make JSON the clear owner of rich payload while `fst` remains for thin
  relationships, search projections, and status

Done means:

- `entity_id` is the authoritative internal identity key
- `ref_id` remains the surface-facing key for users and scripts
- BibTeX export normally prefers the published identity when available
- digest/full-text workflows can prefer a linked arXiv identity when appropriate
- large `fst` rewrites are no longer the default cost of small updates
- the package architecture matches the design goals at the top of this note

## Summary

The core refactor idea is:

- identity is `entity_id`
- user-facing ids are identities
- `fst` stores relationships and status
- JSON stores rich payload

That is the right model for making `litxr` more scalable without changing its
local-first and human-usable nature.
