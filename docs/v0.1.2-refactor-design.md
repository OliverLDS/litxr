# litxr v0.1.2 Refactor Design

The goal is not another partial hotspot patch. The goal is to finish the
remaining architectural work so `litxr` stops treating large projected `fst`
tables as primary mutable state.

This plan assumes `litxr` remains:

- local-first
- `ref_id`-friendly at the surface
- `data.table` + `fst` + JSON based
- non-R6

## Scope

This document covers the work after `v0.1.0`.

Main themes:

- minimal normalized identity and payload scaffold beginning in `v0.1.2`
- retirement of legacy wide `delta` projection semantics
- table-local append shards for expensive derived stores only, especially research-schema digests and embedding artifacts
- direct keyed ingest into `ref_arxiv.fst` and `ref_doi.fst` without a delta/cache layer
- incremental or deferred entity refresh
- keyed and by-reference `data.table` refactors
- streaming embedding search
- runtime compatibility projections instead of heavy mutable wide indexes

Terminology rule:

- **delta** refers only to legacy wide projection delta files being retired.
- **append shard**, **table-local append log**, or **narrow write shard**
  refers to the new narrow implementation mechanism for derived stores.
- **pending** is reserved for unresolved bibliographic input state, especially
  `ref_local_pending.fst` and `key_type = "local_pending"`.
- New code should not use `delta` or `pending shard` to describe the new
  narrow append mechanism.
- The append-shard mechanism is allowed only where direct keyed upsert is too
  expensive or structurally unsuitable, especially embedding matrices and
  research-schema digest generation. It is not recommended for normal
  bibliographic ingest into `ref_arxiv.fst` or `ref_doi.fst`.

## Core Conclusions From The Three Notes

The notes agree on five points:

1. Identity state, bibliographic payload, embeddings, digests, and derived
   status/read-model tables should be physically separated.
2. The current codebase still has large architectural bottlenecks even after
   earlier local optimizations.
3. The biggest remaining costs come from write amplification and full-table
   rebuild behavior, not from a few slow scalar expressions.
4. `data.table` is not yet being used in the most valuable way in the hot
   paths: by-reference mutation, keyed joins, grouped aggregation, and buffered
   bind-once patterns.
5. Compatibility projections should remain available for current scripts, but
   should no longer be authoritative mutable storage.

Additional architectural conclusions should be treated as explicit design rules:

6. The wide project references projection must stop being the object that is
   rebuilt first and filtered second. Filtering must happen first on narrow
   canonical ids. Wide rows may be materialized only at the end for the small
   matched result set.
7. The old wide projection delta mechanism must disappear as a storage
   semantic. If append logs remain, they may exist only as narrow table-local
   implementation details for expensive derived artifacts, especially schema
   digests and embeddings, and must never be treated as canonical projection
   truth.
8. `ref_id` is a user-facing selector, not the canonical identity object. The
   target architecture should not maintain a general local alias table that maps
   user-facing `ref_id` to arXiv/DOI. A route function should normalize and
   resolve `ref_id` to a canonical `arxiv_id` or `doi` whenever possible.
9. `ref_arxiv.fst` and `ref_doi.fst` are authoritative bibliographic payload
   tables, not merely routing indexes.
10. Wide compatibility projections are runtime compatibility shapes. They
    should not be stored as normal local files in the target architecture.
11. Normal ingestion of new bibliographic records into `ref_arxiv.fst` and
    `ref_doi.fst` should use direct keyed upsert into those authoritative
    payload tables. It should not create or depend on a delta cache.

## Non-Negotiable Invariants

These rules come from the normalized storage note and should be treated as hard
constraints for the refactor.

### Identity And Routing Invariants

- `ref_id` remains the surface-facing selector accepted by user-facing APIs.
- `ref_id` is routed by function, not by a general local alias table.
- `entity_id` is the internal grouping key for one conceptual work.
- `arxiv_id` is unique when non-missing.
- `doi` is unique when non-missing.
- Missing ids are represented as `NA_character_`, never the string `"N/A"`.
- At least one of `arxiv_id` or `doi` must exist for a canonical entity row.
- Rows without arXiv/DOI are allowed only in `ref_local_pending`, not in
  canonical `ref_entities`, until resolved.

### `ref_id` Routing Rule

A user-facing `ref_id` should be normalized by a simple route function before
any storage lookup.

The route function should:

1. normalize whitespace and obvious prefixes;
2. recognize arXiv ids using fixed arXiv id patterns, including modern ids such
   as `NNNN.NNNNN` and version suffixes where applicable;
3. recognize DOI-like values using DOI normalization rules;
4. return a narrow canonical key candidate:
   - `key_type = "arxiv_id"`, `key_value = <arxiv_id>`; or
   - `key_type = "doi"`, `key_value = <doi>`; or
   - `key_type = "local_pending"`, `key_value = <ref_id>` when unresolved;
5. avoid creating or depending on a persisted `ref_id -> arxiv_id/doi` lookup
   table in normal target code.

Current scope rule:

- for current user-facing scripts, `ref_id` means only arXiv-like ids or
  DOI-like ids
- this is an intentional restriction to avoid inflating local routing tables
- if future workflows need additional identifier classes, user-facing scripts
  may change their parsers from generic `--ref-id` into strict typed parsers
  for each canonical key class instead of expanding local alias tables
- the default architectural response to new identifier classes is stricter
  parser contracts, not wider persisted routing tables

Implementation note:

- arXiv id parsing should be centralized in one route helper rather than
  reimplemented in multiple read/write functions.
- A small in-memory memoized cache is acceptable inside one call stack, but the
  target architecture should not introduce a persistent `index/ref_aliases.fst`
  as a general solution.

### Payload Coverage Invariants

- `ref_arxiv.fst` is the authoritative arXiv bibliographic payload table.
- `ref_doi.fst` is the authoritative DOI/Crossref bibliographic payload table.
- If a canonical entity row contains `arxiv_id`, the arXiv payload must exist in
  `ref_arxiv.fst`.
- If a canonical entity row contains `doi`, the DOI payload must exist in
  `ref_doi.fst`.
- Linking DOI to arXiv is an identity operation, not a payload-merge shortcut.
- Payload tables should contain scalar persisted fields directly.
- List-like payload fields must not be stored as canonical `fst` list columns.

### List-Column And Rich-Payload Invariants

- List columns are not canonical persisted `fst` fields.
- Persist list-like fields as JSON strings, JSON payload blobs, or normalized
  child tables.
- Reconstruct list-column shapes only at API/export boundaries.
- `authors_list`, `authors_list_json`, categories, affiliations, and similar
  nested fields must not become hidden canonical truth inside wide projection
  caches.

### Derived Store Invariants

- Embeddings are optional derived state, not core identity state.
- Research-schema digests are optional derived state, not core identity state.
- Embedding and research-schema digest stores may use append shards or narrow
  write shards before a future one-time compaction, because they are expensive
  derived artifacts rather than primary bibliographic ingest tables.
- `ref_arxiv.fst` and `ref_doi.fst` should not use a delta cache for normal
  ingestion of new records. They should be updated through direct keyed upsert
  or an explicit batch rewrite/upsert path.
- Thin read models must be rebuildable from authoritative entity, payload,
  digest, embedding, and project/collection state.
- Derived read models do not need to have a one-to-one file-count relationship
  with authoritative stores.

## Target Logical Storage Model

The normalized design note implies the following target logical model.

### Authoritative Stores

These stores are authoritative. Normal reads and writes should converge toward
these files and away from wide mutable projections.

#### `ref_entities.fst`

Canonical entity table.

Key semantics:

- one row per conceptual work
- keyed by `entity_id`
- `arxiv_id` unique when non-missing
- `doi` unique when non-missing

Expected columns:

- `entity_id`
- `arxiv_id`
- `doi`
- `created_at`
- `updated_at`

Important rule:

- `ref_id` should not be a required canonical column here.
- If a caller supplies `ref_id`, route it to `arxiv_id` or `doi` first.

#### `ref_arxiv.fst`

Authoritative arXiv bibliographic payload table.

Key semantics:

- keyed by `arxiv_id`
- contains arXiv-derived bibliographic payload
- may contain scalarized fields such as title, abstract, publication date,
  primary category, version, source timestamps, and payload hash
- list-like fields must be encoded or normalized, not persisted as `fst` list
  columns
- normal ingest of new arXiv payload rows should use direct keyed upsert into
  this authoritative table, not a delta cache or projection delta

#### `ref_doi.fst`

Authoritative DOI/Crossref bibliographic payload table.

Key semantics:

- keyed by `doi`
- contains DOI/Crossref-derived bibliographic payload
- may contain scalarized fields such as title, abstract, journal, publisher,
  publication date, source timestamps, and payload hash
- list-like fields must be encoded or normalized, not persisted as `fst` list
  columns
- normal ingest of new DOI/Crossref payload rows should use direct keyed upsert
  into this authoritative table, not a delta cache or projection delta

#### `schema_arxiv.fst`

Research-schema digest table for arXiv records.

Key semantics:

- keyed by `arxiv_id`
- contains research-schema digest only
- derived from authoritative arXiv payload plus schema generation state
- may use append shards or narrow write shards for generated digest rows before
  a future one-time compaction into this table

#### `schema_doi.fst`

Research-schema digest table for DOI records.

Key semantics:

- keyed by `doi`
- contains research-schema digest only
- derived from authoritative DOI payload plus schema generation state
- may use append shards or narrow write shards for generated digest rows before
  a future one-time compaction into this table

#### `embed_arxiv_abstract/`

Embedding store for arXiv abstracts.

Physical storage:

- narrow metadata table keyed by `arxiv_id`
- matrix shards for embeddings
- shard manifest

Append-shard rule:

- embedding generation may naturally write new matrix/metadata shards first
- future one-time compaction may merge these shards into larger optimized
  shards or a compacted metadata table
- this is allowed because embeddings are expensive derived artifacts, not
  primary bibliographic ingest state

#### `embed_doi_abstract/`

Embedding store for DOI abstracts.

Physical storage:

- narrow metadata table keyed by `doi`
- matrix shards for embeddings
- shard manifest

Append-shard rule:

- embedding generation may naturally write new matrix/metadata shards first
- future one-time compaction may merge these shards into larger optimized
  shards or a compacted metadata table
- this is allowed because embeddings are expensive derived artifacts, not
  primary bibliographic ingest state

### Pending Input Store

#### `ref_local_pending.fst`

Temporary holding table for unresolved records.

Purpose:

- hold rows that cannot yet be routed to `arxiv_id` or `doi`
- preserve local-first ingestion without polluting canonical entity state
- allow later resolution into `ref_entities`, `ref_arxiv`, and/or `ref_doi`

Rules:

- rows here are not canonical entities
- rows here should not be treated as resolved bibliographic truth
- rows here should not participate in normalized DOI/arXiv linking until they
  are resolved

### Derived Thin Read Models

These are read models, not authoritative stores. They intentionally do not
mirror the authoritative store list one-to-one.

Important clarification:

- Authoritative stores answer: **where does truth live?**
- Derived thin read models answer: **what narrow routing/status summary makes
  reads cheap?**
- Therefore the number of read-model files does not need to equal the number of
  authoritative files.
- One read model may summarize multiple authoritative stores.
- Some authoritative stores may need no separate read model.
- Some old read models may be tolerated only during migration.

Target derived thin read models:

- `index/entity_collections.fst`
  - narrow membership/routing table between canonical ids and collections or
    projects
  - should contain only keys and minimal collection/project routing fields
- `index/entity_status.fst`
  - narrow status table for payload, schema, embedding, artifact, and freshness
    status
  - may summarize multiple authoritative stores
- `index/sync_state.fst`
  - narrow sync/checkpoint state for append shards, compaction, source fetches,
    and deferred refresh

Deprecated or migration-only read models:

- `index/ref_aliases.fst`
  - should not be part of the target architecture because `ref_id` routing is
    function-based, not table-based
- `index/entities.fst`
  - if currently used, should become a migration compatibility cache or be
    replaced by direct reads from `ref_entities.fst`

## Compatibility Projection Policy

Some current scripts may still expect wide reference-shaped tables. This
compatibility should be preserved at the API/output-shape level without storing
wide projections as normal local files.

### Target Compatibility Projection Shapes

The following shapes may remain as compatibility interfaces or runtime materialized
outputs:

- collection `index/references.fst` shape
- project `index/references.fst` shape
- project `index/reference_collections.fst` shape

Important rule:

- These are target compatibility **shapes**, not target persisted truth files.
- Normal target code should not write these as local mutable storage.
- A wide compatibility projection may be materialized in memory only after
  narrow canonical filtering.
- A temporary/export/debug file may be produced only by explicit command and
  must be marked non-authoritative.

### Legacy Files Tolerated During Migration Only

The following legacy files may be read only by explicit non-user-facing
migration, repair, audit, or compatibility-diagnosis tooling:

- collection `index/references_delta.fst`
- project `index/references_delta.fst`
- project `index/reference_collections_delta.fst`

Rules:

- These legacy delta files must be read only by explicit migration, repair,
  audit, or compatibility-diagnosis helpers.
- They must not be written as part of the normalized target architecture.
- They must not be treated as canonical truth.
- They must not be the primary read model.
- They should be explicitly retired after compatibility migration is complete.

Transition rule:

- no normal user-facing function is allowed to read legacy wide indexes during
  transition
- before the next version after this design note is implemented, such reads
  become forbidden in normal code paths
- legacy wide indexes may be read only by explicit migration, repair, audit, or
  compatibility-diagnosis helpers
- if a function still depends on a legacy wide index, that function is not yet
  migrated

### Runtime Wide Projection Guardrail

Wide compatibility projections are allowed only as small runtime intermediates.

Rules:

- Always route and filter `ref_id` or other user input to canonical narrow keys
  first.
- Materialize wide rows only for the final matched canonical id set.
- If the final matched id set contains more than the function's configured
  runtime wide projection limit, stop before building the wide projection and
  exit with an informative warning.
- The warning should explain that the caller must narrow the query, request a
  narrow result, or call an explicit export/batch materialization function.
- Normal lookup of one `ref_id` should usually materialize one wide row or no
  row.

Configuration rule:

- `300` is not a universal hard-coded architectural constant
- it is the default input value for functions that need runtime wide projection
- functions that materialize wide rows should expose this as an explicit
  argument or configuration value
- explicit export or batch-materialization functions may choose a larger limit
  or no limit, but that must be opt-in

Example warning text:

```r
warning(
  sprintf(
    "Wide compatibility projection aborted: matched %d canonical ids, which exceeds the configured runtime materialization limit of %d. Filter more narrowly or request a narrow result.",
    n_matched,
    runtime_wide_projection_limit
  )
)
```

## Performance Rules

These rules come from the `data.table` note and should govern implementation
choices.

### Required `data.table` Style In Hot Paths

- Prefer `:=` for grouped or scalar in-place updates.
- Use `data.table::set()` inside tight loops.
- Use `setorder()` / `setorderv()` instead of ordered-copy patterns.
- Use keyed joins or explicit `on=` joins instead of base `merge()`.
- Buffer rows or chunks in lists, then `rbindlist()` once.
- Push row filters into `i` before `by`.
- Avoid repeated `copy()` when row filtering already allocates.
- Use lookup joins or `fcase()` instead of long scalar condition chains.

### Forbidden Hot-Path Patterns

These are the patterns the diagnosis note flagged and this refactor should
remove from the core paths.

- read/merge/rewrite whole legacy delta files on every append
- eager full entity refresh after small writes
- per-key `lapply()` over one-row `data.table` slices inside upsert core
- row-by-row `file.exists()` and row-by-row JSON hydration in project reads
- full-corpus embedding load and score for normal top-N search
- base `merge()` / `split()` / `lapply()` pipelines in entity builders
- repeated small random one-row FST reads for medium or large key sets
- rebuild-wide-then-filter execution order for project reference lookup
- wide projection delta files treated as canonical mutable state
- persisting wide compatibility projections as normal local storage
- building runtime wide projections for more than the function-configured/default runtime wide-projection limit
- adding a delta/cache layer for normal ingest into `ref_arxiv.fst` or
  `ref_doi.fst`

## Current Bottlenecks To Eliminate

The implementation should explicitly close the remaining bottlenecks identified
by the diagnosis note.

- legacy delta rewrite-on-append
- eager `.litxr_entity_index_maybe_refresh()` behavior
- scalar/key-loop `.litxr_upsert_records()`
- non-streaming embedding search
- dense score materialization in optional query-score paths
- project hydration that is improved but not yet fully keyed and batched
- entity builders using `merge()`, `split()`, `lapply()`, and repeated
  existence checks
- compatibility lookup paths that still do one-row random FST reads for larger
  key sets
- wide project reference materialization happening before canonical narrow-key
  filtering
- wide delta semantics surviving as if they were part of normalized truth
- ambiguity between authoritative stores and derived thin read models

## Primary Read-Path Rule

This rule should control all future lookup and reporting refactors.

Normal execution order should be:

1. normalize the input `ref_id` or key;
2. route it to a canonical key candidate such as `arxiv_id` or `doi`;
3. resolve to `entity_id` only when entity-level grouping/linking is required;
4. filter on narrow canonical ids first;
5. join only minimal routing/status fields next;
6. materialize wide bibliographic rows only for the small matched set;
7. abort runtime wide materialization if the matched set is larger than the function-configured/default runtime wide-projection limit;
8. reconstruct list-column/API shapes only at the boundary when required.

The inverse order is explicitly disallowed:

- rebuild a wide project references table
- scan or filter that wide table
- hydrate or join rich fields before narrow filtering
- persist the rebuilt wide table as normal local state

That old order is the main thing the refactor is intended to remove.

## Append Shard Semantics Rule

Under the normalized architecture, legacy delta is not a logical data model
concept.

Canonical truth should live in:

- normalized authoritative entity state
- normalized authoritative payload state
- normalized derived stores where applicable
- narrow read models rebuilt from authoritative state

Normal bibliographic ingest should be direct:

- new arXiv payload rows should be keyed-upserted into `ref_arxiv.fst`
- new DOI/Crossref payload rows should be keyed-upserted into `ref_doi.fst`
- normal payload ingest should not write a delta cache, temporary payload cache,
  or wide projection delta before reaching the authoritative payload table

If append-only logging remains for performance, it must be called an append
shard, table-local append log, or narrow write shard. In the
target architecture this mechanism is allowed mainly for expensive derived
artifacts, especially schema digests and embedding stores, where one-time
future compaction is useful.

It must follow all of these rules:

- it is table-local
- it is narrow
- it is append-only or shard-based
- it compacts into the same derived store or derived-store metadata table
- it is never a wide project-reference projection delta
- it is never the authoritative source of identity truth
- it is never the normal ingest path for `ref_arxiv.fst` or `ref_doi.fst`
- it is never relied on as the semantic equivalent of the final projection
- it is never named as a new `delta` mechanism in target code

Examples of allowed implementation details:

- append shards for `schema_arxiv.fst`
- append shards for `schema_doi.fst`
- matrix/metadata shards for `embed_arxiv_abstract/`
- matrix/metadata shards for `embed_doi_abstract/`

Examples of disallowed target architecture:

- `project references delta` as canonical compatibility truth
- `project reference_collections delta` as canonical compatibility truth
- write paths that update a wide projection delta and treat it as the primary
  source for normal reads
- new target files named `references_delta.fst`
- `ref_arxiv_delta.fst`, `ref_doi_delta.fst`, or similar delta caches for
  normal bibliographic payload ingest

## Refactor Strategy

This plan is staged. The normalized scaffold should begin immediately in
`v0.1.2`, because later read/write optimizations need stable canonical storage
semantics.

The practical sequence is:

1. add a minimal normalized identity/payload scaffold;
2. retire legacy wide delta semantics and replace only derived-artifact
   performance behavior with narrow append shards;
3. move reads onto narrow identity-aware filtering before any wide
   materialization;
4. make table-local append and entity-update behavior cheap;
5. move remaining writes onto normalized stores;
6. keep compatibility projection shapes as runtime outputs only;
7. demote or remove legacy heavy paths.

This avoids the earlier conflict where normalized authoritative stores were
specified as a late milestone even though earlier milestones depended on their
semantics.

## Implementation Rules For Codex

Use these as working constraints during the refactor.

- Prefer `:=`, `set()`, keyed joins, grouped `by =`, and `rbindlist()`.
- Avoid base `merge()` in hot paths unless there is a compelling compatibility
  reason.
- Avoid repeated `split()` + `lapply()` pipelines when grouped `data.table`
  operations can express the same logic.
- Avoid repeated `file.exists()` in per-row logic.
- Avoid repeated `copy()` unless mutation isolation is required.
- Keep compatibility projection shapes runtime-only unless an explicit export or
  migration command requests a file.
- Keep rich/list-like payload fields encoded, JSON-backed, or normalized rather
  than canonical `fst` list columns.
- Preserve user-facing CLI behavior unless there is an explicit decision to
  change it.
- Do not create a persistent `ref_id -> arxiv_id/doi` alias table in the target
  architecture.
- Do not use `delta` as the name for new append-shard behavior.
- Do not add delta/cache files for normal ingest into `ref_arxiv.fst` or
  `ref_doi.fst`; use direct keyed upsert instead.
- Allow append shards or narrow write shards for schema digest and embedding
  generation where future one-time compaction is beneficial.

## Practical Definition Of Success

This refactor is successful when all of the following are true.

- A minimal normalized identity/payload scaffold exists from `v0.1.2` onward.
- A small bibliographic append goes directly into `ref_arxiv.fst` or
  `ref_doi.fst` without a delta/cache layer.
- Derived schema/embedding writes may use append shards and future one-time
  compaction.
- A small derived append no longer rewrites a whole legacy delta table.
- A batch sync no longer rebuilds entity indexes after every small write.
- Upsert behavior is join-first instead of row-slice-first.
- Normal embedding search is shard-streaming.
- Entity builders are keyed and grouped instead of `merge()`/`split()` heavy.
- Compatibility lookup output remains stable for current scripts.
- Wide compatibility shapes are runtime materializations, not normal persisted
  local files.
- Runtime wide materialization aborts above the function-configured/default runtime wide-projection limit.
- Normalized storage semantics are explicit and enforced.
- Rebuild and audit paths are explicit instead of hidden in ordinary reads.
- Derived thin read models are understood as read models, not one-to-one mirrors
  of authoritative stores.
